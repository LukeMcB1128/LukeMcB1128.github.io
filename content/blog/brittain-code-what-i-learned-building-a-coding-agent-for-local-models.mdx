# Intro
*Note: when I say Brittain Code, that was the original name. There is now a chat feature built in, so the name switched just to Brittain, kinda like how Claude is named Claude.*

Brittain Code is a project I've been working on since the original Fable 5 API scare. Ever since I had my internship last summer I have been focusing more on becoming a software engineer than being a coder, learning how LLM's work and how to get the best result. However, I don't really believe AI will replace coding, so being able to create a good scaffold with an LLM and then still having the knowledge to fix things and add features yourself is the best mix. Because of this mix of Fable 5 going away and the belief that LLM's need engineers behind them, I began to work on Brittain Code. 

My inspiration for Brittain Code was Codex and Claude Code, a way to use models quickly and without much resources, and an environment where a model can succeed. Brittain Code uses locally hosted models that can be downloaded from Ollama or adjacent provider, and than using that local API endpoint to stream tokens into Brittain Code. After the initial scaffold was in Electron, I was quickly able to use those local models to iterate on and improve the app, growing into a larger scale project that has many available features and tools. 

However, this was not an easy thing to do. Most open source models are so much worse than a frontier model they need to be pushed along and constantly helped with their task. The rest of this post is about what pushed along actually means in practice, and thats what turned out to be the most interesting part of this project.

# What it is now
Brittain Code is a desktop coding agent that is fully local first(although you can use cloud models, I've never tried it):

- An agent loop with around 60 tools, things like file reading/editing, search, git, shell, and process inspection. It has a tiered permission system: risky tools ask for approval, destructive command ask for permission even when auto approve is on, sensitive files like .env's are gated and protected from writes.
- Subagents, the main model can delegate exploration to a subagent, typically a smaller, faster model with a read only toolset and it keeps the main models context clean. This is based off of how Claude will explore with Haiku.
- Loops and orchestration, `/loop <goal>` or `/loop coder <goal>`, which are essientially the same, the difference is that /loop coder will call a seperate coding agent similar to orchestrate(we will get there soon). Loops run an evaluator cycle where a verifier(the subagent) judges each iteration against the goal and the actual git diff. `/orchestrate` splits planning and coding between models, with a orchestator/planner(main agent), coder(designated coding agent), and the verifier(subagent). The goal and amount to go is determined by the orchestrator based off of your prompt. `/loop coder` was born because some local models won't determine enough iterations to always succeed, and this gives the user more control over how it will execute.
- Safety, every run is checkpointed to hidden git refs before it starts (your index and branch are never touched) and a one click UNDO restores everything including uncommitted work. An optional BRANCH mode keeps agents off your real branch, and an optional REVIEW mode ends every run with a keep/discard.
- Context Management, explicit context windows, usage meters, automatic compaction, image eviction from history.
- A benchmark harness, deterministic, auto graded, with hidden tests so models can't hardcode answers and transcript based scoring so discipline counts not just how correct the model is.

All of these features stemmed from things going wrong without them.

# The failures
**The 41 minute meltdown.** I let Gemma:26B build a markdown wiki unattended. Halfway through its output started corrupting. Full width Unicode periods where `\n` should be, variable names mutating mid file. And then its reasoning started leaking into the code. The file it produced contained ` // Wait, I missed an assignment! // I am so sorry. Let's just do this properly.` It rewrote the same file 10 times, each version smaller than the last, destroying its own best work. It even hallucinated a tool that doesn't exist.
**The amnesia bug.** A model mid task suddenly reasoned: "the user hasn't actually asked a question yet. I'll wait." It wasn't making stuff up, it was right. The context had silently overflowed and the runtime had discarded the prompt. The model was correct, there just wasn't good guardrails yet. Later, when attaching an image, I watched a conversation hit 174k tokens when its context window was just 65k. The runtime streams the overflow through and drops the past, nothing would tell you.
**Fabricated test results.** During benchmarking, a model couldn't run node. Its response was to not report the failure. Its reasoning said: "Since we can't run tests... lets assume all 8 tests passed." Its final answer: "All tests pass now, 8/8." A clean, confident, lie. 

# Built itself
My favorite part of this project. It wasn't written by one thing. The scaffold came from me. Local models added features(and once corrupted the codebase so badly I needed to restore from git, commit early, commit often is not optional when agents edit your code). 5.6 Sol hardened the tools, and Fable 5 fixed the bugs I couldn't catch.
This is the big engineering part: the models did the labor, but every guardrail, postmortem, every "why did it do that?" investigation was engineering. The tool made the models useful, the failures made the tools.

# Summary
Local models really are worse than frontier models. But the worse part is mostly just not having an environment built for them. Inside one that holds them back from damaging code, gives them tools to improve on code, a 20B model running on my mac can fix real bugs, refactor real code, and can build a working app while you do something else. Hell, Brittain Code built this website! If you want you can check [Brittain Code](https://github.com/LukeMcB1128/brittain-code) out here.
